## Principles (non-negotiable)

1. **On-device only.** Models ship as app assets (or first-run download), load into RAM, run in the
   existing DSP isolate via LiteRT/TFLite. No audio ever leaves the phone. Cloud LLMs are research
   tools, never runtime components.
2. **Seam + fallback.** Every ML component sits behind an interface next to the current pure-Dart
   implementation (`StrumDirectionClassifier`: heuristic ↔ TFLite, flag-selected). The heuristic
   stays the DEFAULT until the model beats it in the property gates AND on the real guitar.
3. **HORIZON acceptance.** Synthetic F1 is never "done". The final gate for flipping any ML default
   is the user's real-guitar APK test.
4. **This box can't run TF** (ARM64) — training runs on x86_64 GitHub Actions CI or Colab.
   Everything trainable must be reproducible from committed code (`ml/`).
5. **Feature parity is a tested contract.** The Dart front-end (log-mel) must match `ml/features.py`
   bit-for-tolerance on golden fixtures generated by the Python pipeline — a drifted front-end
   silently destroys a trained model.

**GAME-CHANGER (verified 2026-07-12):** the ISMIR-2025 dataset/code/checkpoint are now PUBLIC —
`github.com/Klangio/guitar-strumming-transcription`, **Apache-2.0** (API-verified: licence + repo
exist, ~770 MB, dataset in-repo). Contents: **56 recording sets** (90 min real audio from 3
guitarists, each take as `_phone.wav` — our exact deployment condition — + `_line.wav` + IMU CSV +
`.strums` direction labels), training scripts, and a pretrained checkpoint (`f1=0.8225`, Git-LFS).
Also `Klangio/KLANGIO-GST-MM-T` (+5 min, different rig) for held-out eval.

1. **Acquire + verify:** clone the dataset (this box CAN hold 1 GB), adapt
   `ml/prepare_dataset.py` to read `.strums` labels; keep GST-MM-T strictly as eval.
2. **Synthetic data:** joint-train on synthetic+real — their ablation shows joint training BEATS
   sequential pretrain→finetune, and **sample-based/VST-rendered strums ≫ pure Karplus-Strong**
   (synthetic-only F1 89.8 % vs 66 % on the sibling task). Our KS `ml/synth.py` stays for smoke
   tests/augmentation, not as the main training source. ±6-semitone pitch-shift stays (+14 % rel. up-strum F1).
3. Train on CI/Colab → dynamic-range-int8 TFLite → P1 plumbing → A/B gates → **real-guitar APK
   test** → only then flip the default. Target: match/approach F1-any 92.75 % (their hybrid), on-device size.
4. **User recording session is now OPTIONAL** — domain adaptation to the user's guitar/room only
   (Sensor Logger on Wear OS, 3 sync claps, cross-correlation refine; ~1–2 h target). Not a blocker.
5. Up-strum is the intrinsically weak class: class-weight the loss; report per-class accuracy honestly
   in the Progress "strum accuracy" metric only from measured data.

## Research notes (folded in as results land)

- ✅ **LiteRT/tflite_flutter (researched 2026-07-12):** use **`flutter_litert`** (v3.5.0, tflite_flutter-compatible
  Interpreter API, bundled LiteRT 1.4.2, 16 KB-page compliant — `tflite_flutter` 0.12.1 has an OPEN Play-compliance
  16 KB issue #287); **neither pulls win32** (repo constraint safe). Stateful GRU = **explicit-state pattern**
  (kws_streaming "external state"): export `(frame[1,1,nMels], h_in[1,128]) → (softmax[1,2], h_out[1,128])`,
  feed `h_out`→`h_in` between `runForMultipleInputs` calls; reset = zero the buffer. Delegate: NONE — plain CPU
  + default XNNPACK, `threads=1` (GPU delegate lacks GRU ops → graph split → slower; NNAPI deprecated in
  Android 15). Expected ≪1 ms/invoke at 100 Hz (streaming-KWS evidence, arXiv 2005.06720). Quantization:
  **dynamic-range int8 (weights-only)** — safe for the GRU (full-int8 recurrent conversion is a known TFLite
  pain point), no calibration set, ~4× size cut. Property-gate idea: N random frames through the stateful
  model == the non-streaming reference.
- ✅ **Datasets (researched + API-verified 2026-07-12):** Klangio released everything (see Phase 2).
  Correction to chunk 015: the paper used **90 min real + 4 h VST-synthetic**, not 94 h; labels came from an
  ESP32+QMI8658 wrist module with semi-automatic GUI alignment. Joint synth+real training beat sequential
  finetuning. No other direction-labeled corpus exists as of mid-2026.
- ✅ **Pitch models + CQT (researched 2026-07-12):** licence verdict — **SPICE** is the only zero-risk drop-in
  (Apache-2.0, official TFLite, 8.7 MB, voice-trained → validate on guitar); **PESTO** is technically best
  (130k params, streaming VQT, <10 ms) but **LGPL-3.0** + PyTorch/ONNX-only → compliance dance in a
  closed-source app (keep as swappable asset if ever used); **CREPE-tiny** (MIT, ~2 MB, convertible) is the
  clean-licence fallback we'd convert ourselves. **CQT in Dart: feasible** — 24 bins/oct × 6 oct @16 kHz ≈ one
  8192-pt FFT + sparse multiply ≈ 1–3 ms/frame on a mid-range phone (fftea benchmarks extrapolated) = 10–30 %
  of a core at 10 ms hop; NO existing Dart CQT package → implement Brown–Puckette 1992 in-house over fftea
  (~100 lines, kernel precomputed, sparsity threshold ~0.005, optional per-octave FFT sizes).

The live arrow keeps the heuristic verdict at ~70 ms (r164: 39 % on real
guitar — the confidence tier must stay honest about it). The CRNN needs the

~170 ms AFTER today's verdict. Options, in preference order:
1. **Delayed refine:** show the heuristic arrow at 70 ms, run the CRNN window

   emit a corrected `LiveFrame` (scorer consumes the refined direction — the
   Learn hit windows are ±120 ms so a 240 ms correction still lands inside
   the scoring window at ≤~150 BPM). UX risk: visible arrow flips — needs
   on-device feel-testing (the real-guitar gate).
2. **Retrain shorter-window model** (POST 7 ≈ 70 ms): eval accuracy cost

   box; worth an experiment round.

   fail the feel test.
